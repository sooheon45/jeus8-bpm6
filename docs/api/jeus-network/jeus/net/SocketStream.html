Class SocketStream

  jeus.net.SocketStream

jeus.io.handler.StreamContentReceiver
public class SocketStream
extends java.lang.Object
implements jeus.io.handler.StreamContentReceiver
SocketStream은 물리적 소켓과 일대일 대응되며 소켓을 관리하고 쓰기 작업을 하는데 주로 사용된다. 내부적으로는 StreamHandler로 부터 읽기가 완료된 데이터를 받아 메시지로 변환시키고 이 메시지가 NetworkControlPacket인 경우 이에 대한 처리도 담당하고 있다. 또한 SocketStream은 안전한 소켓 Close를 위해 Close 상태 관리도 담당하고 있어 JEUS NETWORK LIBRARY에서 아주 중요한 역할을 담당하는 클래스이다. 이 class의 역할은 다음과 같다.

1. 현재 connection의 정보 2. 현재 connection의 제거 기능 3. 현재 connection으로 data를 보내는 통로

isCrossConnectionCloseReceived

public boolean isCrossConnectionCloseReceived()
SocketStream이 CrossConnection에대한 close 메시지를 받았는지 확인한다.

cross connection에 대한 고찰 1. 한 노드에서 다른 노드로 접속을 하고자 하는 thread 간에 하나만 connect를 하도록 하는 것은 Endpoint의 getSocketStream() method에서 target당 하나의 ticket을 받아서 그 ticket에 lock을 걸고 connection을 맺고 나머지 thread는 그 ticket을 얻기위해 대기하는 식으로 해서 하나의 thread만 connect를 수행하도록 처리하였다.

2. 두 노드가 서로 동시에 connect를 수행하는 시나리오에서는 자기 노드의 listen socket ID와 상대 노드의 listen socket ID를 비교해서 connect를 누가 할 지 정한다. connect를 수행하지 말아야 할 노드에서 connect를 수행했다면 현재, 상대 노드로의 접속을 시도하지 않는 상황이라면 그냥 접속을 허용한다. 또는 connect를 해도 되는 노드라면 접속을 허용한다. 이 경우 접속을 허용하기로 결정한 후에 이 노드로 connect를 하려는 thread가 생긴다면 이 thread는 접속을 허용하기로 한 socket stream을 등록할때까지 대기해야 한다. 따라서 Endpoint의 getSocketStream에서와 같이 ticket을 발급받고 socket stream을 등록한다. 이때 주의할 점은 접속을 허용하기로 결정하기 전에 connectionManager의 lock을 잡고 ticket을 발급받아야 connect를 수행하는 thread와의 ticket 발급 경쟁에 의한 race condition이 발생하지 않는다.

접속을 허용하지 않았다면 그 접속을 시도한 connect는 establish가 되지 않았고 OP_CLOSE에 의해 stream이 닫겨져 있는 것으로 지금 현재 자신에게 target 노드가 접속을 시도중이라는걸 알수 있다. 이 경우에는 connectionManager를 통해 해당 socket stream이 accept될 때까지 wait()하다가 accept가 되면 이를 사용하면 된다. (Connector 참고)

3. 만약 접속을 허용하지 않고 자신이 접속하다가 여기서 error가 나는 경우 접속을 기다리는 connector가 계속 기다리게 된다는 문제가 있다. 이런 경우에는 timeout을 설정하고 max try count만큼만 시도하고 exception을 던지도록 하였다.

Returns:
SocketStream이 CrossConnection에대한 close 메시지를 받았는지 여부
